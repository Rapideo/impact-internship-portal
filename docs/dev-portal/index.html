Project Overview

The IMPACT Internship Assessment Portal is a web app for Indiana’s IMPACT internship program. It tracks intern intake, multi-phase competency assessments, intern self-assessments, and 90-day / 180-day employment outcomes. A clickable prototype is locked; the production rebuild is fully planned and now beginning execution.

Supabase Deep-Dive

Supabase is the back-end-of-choice for this project. It deserves its own tab because the security model rests on one of its features (Row-Level Security) doing real work at the database layer.

1. What it is

Supabase is a managed back-end-as-a-service built on top of Postgres. The product bundles a hosted Postgres database, an Auth service, file Storage, Realtime, and Edge Functions. Pricing is per-project with a free tier suitable for development.

The crucial framing: Supabase is not a custom database. The database itself is plain Postgres — the same Postgres you would run on any host. Anything you can do in Postgres, you can do in Supabase. The difference is Supabase manages the hosting, backups, and the surrounding services.

2. What we use

Postgres

The 15-table application database, accessed via Drizzle ORM with strict typing.

Auth (email + password)

Admin and employer login. Anonymous interns never authenticate — they are identified by First Initial + Last Name + Employer + Cohort at submission time.

Row-Level Security (RLS)

Permission rules enforced inside the database. Every query carries the user’s JWT claims; Postgres decides at the row level whether the query is allowed. See section 3.

Storage (not yet)

File uploads aren’t in v1 scope. If interns ever upload a resume, this is where it will live.

Edge Functions (not yet)

We use Netlify’s edge for SSR. If we later need Postgres-co-located server logic, this is the option.

Realtime (not yet)

No live-updating views are planned. The assessment forms are submit-once, immutable — no realtime needed.

3. Why this matters for security

The system has three actor classes — admin, employer, and anonymous intern — and each can only see a strict subset of the data. The naive way to enforce that is in application code: every query gets a WHERE employer_id = ? clause, and the app-code author has to remember to add it every single time.

Row-Level Security moves that enforcement into the database itself. The policies say things like “an employer can only SELECT interns where interns.cohort_id IN (SELECT id FROM cohorts WHERE employer_id = auth.jwt().employer_id)”. The database refuses any other query — even if app code is buggy or compromised.

This is the load-bearing reason Supabase was chosen. An app-code bug cannot leak one employer’s interns to another, because the database itself will not return them. The only way to bypass RLS is to use the service-role key, which never leaves the server (file convention: .server.ts suffix, enforced by the bundler).

4. Three environments (dev, test, prod)

The system uses three independent Supabase projects, one per environment. Each has its own URL, keys, schema, and data. Running the same Drizzle migrations against all three keeps the schemas aligned; the data stays isolated.

impact-dev

Development

Local developer machines (.env.local) and Netlify deploy previews. Seed data; wipeable anytime.

impact-test

CI / Testing

GitHub Actions integration + RLS policy tests run here. Test fixtures wipe before each CI run.

impact-prod

Production

Real intern data. Daily Supabase backups. Only the production deploy of main connects here.

Netlify’s per-deploy-context env var feature does the routing: production deploys read impact-prod credentials; PR deploy previews read impact-dev. A PR can never accidentally touch production data.

Workflow & SDLC

How code moves from a developer’s machine to production. Every change goes through the same path; there are no shortcuts and no direct-to-production pushes.

1. How code reaches production

  1. Branch. The developer creates a short-lived branch off main using a typed name (feat/, fix/, chore/, etc.).
  2. Commit. Each commit message follows the Conventional Commits format. A commit-msg Git hook rejects messages that don’t.
  3. Push + PR. The branch is pushed and a Pull Request is opened against main. The PR template asks the author to link the spec section or plan task and to drop the deploy preview URL.
  4. CI. GitHub Actions runs lint, typecheck, unit tests, integration tests, e2e tests, accessibility checks, bundle-size check, and production build. ~5–8 minutes total.
  5. Review. An AI review (/review) inspects the diff. For security-sensitive areas (auth, RLS, secrets, untrusted input) a second pass (/security-review) runs. Matt does the final eyeball.
  6. Merge. Squash-merge to main. A single commit lands per PR.
  7. Deploy. Netlify watches main and auto-deploys to the live URL within ~90 seconds.

3. Deploy previews

Netlify produces a fresh URL for every PR — https://deploy-preview-N--impact-internship-portal.netlify.app. The bot posts the URL as a comment on the PR within ~60 seconds of opening. This is what reviewers (and Matt’s management) click to visually walk through a change before merge. The preview is the actual production build of the PR branch — not a mock.

4. What cannot happen on main

  • No direct pushes — even from admins.
  • No force pushes.
  • No branch deletion.
  • No merge without all CI checks green.
  • No merge with unresolved review conversations.
  • No non-linear history — squash-merge only, no merge commits.

5. Secrets and security

The application needs ~9 secrets (database URL, Supabase keys, session signing key, email API key, error-tracking tokens). None of them are committed to the repo. Three places hold them:

  • Local development — each developer has a .env.local file that is in .gitignore from day one.
  • CI — the same names live in GitHub Secrets, scoped to the repo. The workflow injects them as environment variables for test runs.
  • Production — the same names again, this time in Netlify’s environment-variable settings. They are injected into the running server at request time; they never appear in build logs or source maps.

A canonical list of expected variables lives in .env.example (committed). It contains placeholder values only — copying it to .env.local and filling in real values is the per-developer setup.
