## Finding 1: `color-contrast` on KPI card delta text

**Severity:** serious.
**Routes affected:** `/admin`, `/employer`.
**Where:** `.kpi-card .kpi-card__delta` — small text inside the dashboard KPI cards.
**Measured contrast:** 2.69 : 1 (foreground `#00A6F6` / background `#FFFFFF`).
**WCAG AA requirement:** 4.5 : 1 for normal text.

Both admin and employer dashboards use a KPI rail whose `__delta` line ("ACROSS 1 COHORTS", "WITHOUT A COMPETENCY", etc.) is rendered in the cyan brand token on a white card surface. The cyan reads great on the dark navy nav/footer but doesn't have enough contrast on light surfaces — and CLAUDE.md actually documents this: "Use `--gold` only on dark surfaces; for light surfaces use `--navy` for emphasis."

**Recommended fix:** Swap `.kpi-card__delta` color from `--cyan` to either `--navy` (3.5:1 wouldn't pass either; would need to check) or a darker accent. Easiest one-line CSS change in `app/styles/admin.css`. Verify with axe locally after the change.

## Finding 2: `document-title` missing on some routes

**Severity:** serious.
**Routes affected:** `/login`, `/admin/interns`.

`/login` — `app/routes/_public.login.tsx` does not export a `meta` function. The page surfaces a "Welcome back." heading via `<AuthShell>` but the `<title>` tag stays at whatever the parent layout set. Easy fix: add `export const meta = () => [{ title: 'Sign in — IMPACT' }]`.

`/admin/interns` is more surprising — `app/routes/admin.interns._index.tsx:15` DOES export `meta` with `title: 'Interns — IMPACT Admin'`. Axe still flags `document-title` on this route, which suggests either:

- RR v7 meta is not surfacing into the `<title>` tag during the initial SSR shell read (possible timing issue with how axe samples the DOM after `page.goto`); OR
- A parent layout's meta is overriding the child route's. Worth a 10-minute investigation in a focused session.

The same pattern likely affects other admin/employer routes that have meta declarations — re-running this spec across more routes would clarify scope.

## Recommended fix sequence (focused session)

1. **Color-contrast quick win** (~15 min). Change `.kpi-card__delta` color to a darker accent. Verify with `--workers=1` run.
2. **Document-title sweep** (~20 min). Add `meta` to every route missing one (start with `_public.login.tsx`, `_public.auth.forgot.tsx`, `_public.auth.reset.tsx`, `_public.auth.accept.tsx`). Investigate why `/admin/interns` is flagged despite its meta.
3. **Expand spec coverage** (~30 min). Add the 8 routes listed above to the baseline spec. Re-run and capture the second-pass findings.
4. **Decide enforcement threshold.** Once known violations are at zero, flip the spec from baseline-mode (log-only) to assert-mode (`expect(counts.serious).toBe(0)`) so future regressions break CI.

## Why this is "baseline" not "audit-complete"

A first axe pass on an existing UI always finds a handful of "serious" issues — typically color contrast, missing labels, focus management. The point of this pass is to *measure* the gap honestly without committing to fix it in the same session. The findings above are a credible starting point, not an exhaustive audit. Manual testing (keyboard-only navigation, screen-reader walk-through) catches issues axe can't see and should follow the automated pass.
